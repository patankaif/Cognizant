# Module 5 – Entity Framework Core 8.0 (C#)

A two-project .NET 8 solution covering every learning objective in
**Module 5** of the DN 5.0 Deep Skilling handbook, built Code-First against
SQL Server.

## Solution layout

```
Module5EFCore.sln
src/
  Module5.Data/                      (EF Core model + DbContext)
    Entities/
      Author.cs
      AuthorProfile.cs                one-to-one with Author
      Book.cs                         many-to-one Author, RowVersion concurrency token
      Genre.cs
      BookGenre.cs                    many-to-many join entity (Book <-> Genre)
    Dtos/
      BookSummaryDto.cs
    AppDbContext.cs                   Fluent API model configuration
    LazyAppDbContext.cs               same model, with UseLazyLoadingProxies()
  Module5.ConsoleApp/
    Program.cs                        menu-driven entry point
    appsettings.json                  connection string
    Demos/
      SeedDataDemo.cs
      CrudDemo.cs
      LinqQueriesDemo.cs
      RelationshipsAndLoadingDemo.cs
      PerformanceDemo.cs
Migrations/
  Manual_InitialCreate_Reference.sql  what `dotnet ef migrations` would produce
```

## How to run

**1. Set your connection string**

Edit `src/Module5.ConsoleApp/appsettings.json` and point
`ConnectionStrings:DefaultConnection` at your SQL Server instance.

**2. Create the database**

You have two options:

- **Recommended — EF Core Migrations (from the solution root):**
  ```bash
  dotnet tool install --global dotnet-ef   # once, if not already installed
  dotnet ef migrations add InitialCreate --project src/Module5.Data --startup-project src/Module5.ConsoleApp
  dotnet ef database update --project src/Module5.Data --startup-project src/Module5.ConsoleApp
  ```
  This generates a `Migrations/` folder inside `Module5.Data` with the
  migration C# files (not included in this zip, since they must be
  generated by the tool against your exact EF Core version) and applies
  them to create the database and tables.

- **Alternative — run the reference SQL directly:**
  Open `Migrations/Manual_InitialCreate_Reference.sql` in SSMS and execute
  it. It creates the same schema by hand, for environments without the
  `dotnet-ef` tool.

**3. Run the console app**
```bash
cd src/Module5.ConsoleApp
dotnet run                 # interactive menu
dotnet run -- all          # seeds data, then runs every demo in order
dotnet run -- linq         # run just the LINQ queries demo
```

Valid keys: `seed, crud, linq, loading, performance, all`

Run `seed` (or `all`, which seeds first) before `linq`, `loading`, or
`performance`, since those demos query the seeded Authors/Books/Genres.

Requires the **.NET 8 SDK** and a reachable **SQL Server** instance
(LocalDB, a full SQL Server instance, or SQL Server in a Docker container
all work).

## Notes

- I don't have a SQL Server instance or the `dotnet-ef` tool available in
  this sandbox, so I couldn't run migrations or execute these queries
  end-to-end. Please run `dotnet ef migrations add` yourself to generate
  the real migration files, and let me know if anything needs adjusting.
- `PerformanceDemo.ConcurrencyTokenAsync` intentionally simulates two
  overlapping edits to the same `Book` (using two separate `DbContext`
  instances) to demonstrate `DbUpdateConcurrencyException` being thrown
  because of the `[Timestamp]` `RowVersion` column — that caught exception
  in the console output is the expected, successful outcome of the demo.
- `LazyAppDbContext` is kept separate from `AppDbContext` deliberately:
  lazy loading is convenient for demos but can cause unexpected N+1 query
  problems in real applications, so it's isolated to one demo method
  rather than used as the default context everywhere.
